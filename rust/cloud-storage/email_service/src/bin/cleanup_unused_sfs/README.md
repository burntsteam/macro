# Cleanup Unused SFS Files

This binary deletes unused SFS files and their corresponding database mappings using the SFS bulk delete API. It streams UUIDs from a file, batches them, and sends bulk delete requests to efficiently remove files and their mappings from the `email_sfs_mappings` table.

## Purpose

Part of a three-step workflow to identify and clean up unused SFS files:
1. `find_used_sfs_ids` → Finds SFS UUIDs referenced in email message bodies
2. `find_unused_sfs_ids` → Compares all mappings against used UUIDs to find orphans
3. **This tool** → Deletes the unused files and mappings

## How It Works

1. Counts UUIDs in the input file (streaming, no loading into memory)
2. Streams UUIDs from the file and groups them into batches
3. For each batch (default: 50 files per batch):
   - Sends bulk delete request to SFS API (`POST /file/bulk-delete`)
   - SFS endpoint deletes files from S3 and removes its own DynamoDB metadata
   - For each successful SFS deletion, removes mapping from PostgreSQL `email_sfs_mappings` table
   - Tracks success/failure for each individual file
4. Processes batches concurrently (default: 5 concurrent requests)
5. Falls back to individual deletions if bulk request fails
6. Logs progress every 100 files
7. Provides detailed summary of deletions at the end

**Key Improvements Over Individual Deletes**:
- **50x fewer API calls**: Deletes 50 files in 1 request vs 50 requests
- **Faster execution**: Concurrent batch processing reduces total time
- **Better error handling**: Per-file results even in bulk operations
- **Automatic fallback**: Falls back to individual deletes if bulk fails

**Memory Efficient**: UUIDs are streamed from file and processed in configurable-size batches. Only the current batch is in memory.

## Environment Variables

### Required
- `DATABASE_URL`: PostgreSQL connection string for the MacroDB database
- `SFS_URL`: The URL for the Static File Service (e.g., `https://static-file-service.macro.com`)
- `INTERNAL_AUTH_KEY`: Internal auth key for SFS authentication
- `DESTINATION_URL_PREFIX`: URL prefix for reconstructing destination URLs
  - Example: `https://static-file-service.macro.com/file/`
  - Used to build full destination URL: `{prefix}{uuid}`

### Optional
- `UNUSED_UUIDS_FILE`: Path to file containing unused UUIDs (default: `unused_sfs_uuids.txt`)
  - Generated by `find_unused_sfs_ids`
- `BULK_BATCH_SIZE`: Number of files to include in each bulk delete request (default: `50`)
  - Maximum is 100 (enforced by SFS bulk delete endpoint)
  - Larger batches = fewer API calls but less granular error handling
- `BULK_CONCURRENCY`: Number of concurrent bulk delete requests (default: `5`)
  - Increase for faster deletions, decrease if overwhelming SFS or database

## Deletion Behavior

For each UUID, the tool performs two operations:

### 1. Delete from SFS
- Calls SFS bulk delete API: `POST /file/bulk-delete` (batches of 50 by default)
- SFS endpoint deletes files from S3 and removes its own DynamoDB metadata
- **Success**: File was deleted from S3
- **Already Deleted** (404): File doesn't exist (counted separately, not an error)
- **Error**: Network issue, auth failure, or other error
- Falls back to individual `DELETE /file/{uuid}` calls if bulk request fails

### 2. Delete from PostgreSQL Database
- Only attempted for files that successfully deleted from SFS
- Executes: `DELETE FROM email_sfs_mappings WHERE destination = '{prefix}{uuid}'`
- **Success**: Mapping row deleted
- **Error**: Database connection issue or permission error

The database cleanup only happens after successful SFS deletion to maintain consistency.

## Safety Features

- **Dry run not available**: This tool permanently deletes files
- **No confirmation prompt**: Assumes you've verified the unused UUIDs
- **Continues on errors**: Individual failures don't stop the entire batch
- **Progress tracking**: Can be interrupted and resumed (just run again with same file)
- **Concurrent limits**: `BULK_CONCURRENCY` setting prevents overwhelming services

## Performance

- **Streaming architecture**: Reads UUIDs from file in 1000-item batches
- **Concurrent deletions**: Configured via `BULK_CONCURRENCY` (default: 5)
- **Memory usage**: Constant, regardless of total file count
- **Typical speed**: 100-500 deletions per second (depends on SFS/DB latency)

Typical performance at different bulk concurrency levels:
- BULK_CONCURRENCY 5 (default): ~250 deletions/sec, conservative
- BULK_CONCURRENCY 10: ~500 deletions/sec, recommended
- BULK_CONCURRENCY 20: ~1000 deletions/sec, aggressive (may overwhelm services)

Note: With bulk operations, each request deletes up to 50 files (BULK_BATCH_SIZE), so performance scales much better than individual deletions.

## Exit Behavior

- **Exit code 0**: Always, even if some deletions fail

## Next Steps

After running this tool:
1. Review the summary statistics
2. Check if any files failed to delete (investigate errors in output)
3. Optionally re-run to attempt failed deletions again
4. Archive or delete the `unused_sfs_uuids.txt` file

## Troubleshooting

### "File does not exist: unused_sfs_uuids.txt"
- Run `find_unused_sfs_ids` first to generate this file

### "No UUIDs to delete"
- The input file is empty
- Run `find_unused_sfs_ids` to generate unused UUIDs
- Or verify the file path is correct

### High SFS failure rate
- Check `SFS_URL` is correct and service is reachable
- Verify `INTERNAL_AUTH_KEY` is valid
- Check network connectivity
- Reduce `BULK_CONCURRENCY` to avoid rate limiting

### High DB failure rate
- Check `DATABASE_URL` is correct and database is reachable
- Verify user has DELETE permissions on `email_sfs_mappings` table
- Check `DESTINATION_URL_PREFIX` matches format in database
- Reduce `BULK_CONCURRENCY` to avoid connection pool exhaustion

### "Already deleted" count is high
- Normal behavior if tool has been run before on same file
- Files may have been manually deleted
- SFS may have already cleaned up some files

### Process is too slow
- Increase `BULK_CONCURRENCY` (try 10, 15, or 20)
- Check network latency to SFS
- Verify database connection pool is adequate

### Process is too fast / overwhelming services
- Decrease `BULK_CONCURRENCY` (try 2 or 1)
- Monitor SFS and database server load

### Want to test without deleting
- **Not recommended**: This tool has no dry-run mode
- Alternative: Manually test with a small file (1-10 UUIDs)
- Copy a few UUIDs to a test file and run with that file

## Safety Checklist

Before running in production:

- [ ] Verified `unused_sfs_uuids.txt` contains only truly unused files
- [ ] Checked percentage of unused files is reasonable (in `find_unused_sfs_ids` output)
- [ ] Confirmed `SFS_URL` and `DESTINATION_URL_PREFIX` match your environment
- [ ] Tested with small batch first (optional but recommended)
- [ ] Have database backup (for peace of mind)
- [ ] Set appropriate `BULK_CONCURRENCY` for your infrastructure
- [ ] Ready to monitor the deletion progress

## Recovery

If you accidentally delete files that shouldn't have been deleted:

1. **Database mappings**: Can be restored from database backup
2. **SFS files**: If you have S3 versioning enabled, files may be recoverable
3. **Prevention**: Next time, spot-check unused UUIDs before running cleanup
